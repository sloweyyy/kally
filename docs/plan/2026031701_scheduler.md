# Scheduler — One-Shot Reminders via Cron — 2026-03-17-01

> Allow OpenCode to schedule a one-shot prompt for future execution in the **same session**. Primary use case: user says "remind me in 2 hours to check the deployment" → OpenCode writes a crontab entry that fires once, resumes the original session, and cleans itself up.

## Context

Thor already has a complete cron system:

- `POST /cron` endpoint on gateway, authed with `CRON_SECRET`
- supercronic container watching `/workspace/cron/crontab` via inotify
- `hey-thor` CLI for crontab entries
- OpenCode has **rw** access to the crontab file
- Crond re-reads the crontab every wake cycle (~1 min)

The cron system handles recurring jobs. For one-shots ("remind me in 2h"), we reuse the same infrastructure with two small additions:

1. **`--key` flag on `hey-thor`** — pass a correlation key so the reminder resumes the original session (same Slack thread)
2. **Marker comments + self-cleanup** — the prompt instructs the agent to remove the one-shot entry after firing

No new endpoints, no new services, no EventQueue abuse.

## Design

### Flow

```
1. Session A (Slack thread) — user: "remind me in 2h to check deployment"
2. Session A writes to /workspace/cron/crontab:
     # ONE-SHOT:abc123
     30 17 17 03 *  hey-thor --key "slack:C06XXX:1710000000" "Reminder: check the deployment status. Then remove the ONE-SHOT:abc123 entry from /workspace/cron/crontab."
3. Crond picks up the new entry (within 1 min)
4. 2h later — crond fires → hey-thor POSTs to gateway with correlationKey
5. Gateway enqueues → runner resumes Session A → agent responds in same Slack thread
6. Agent removes the ONE-SHOT:abc123 lines from crontab as instructed
```

### Why this approach?

- **Zero new services or endpoints** — reuses cron infrastructure entirely
- **Same-session continuity** — `--key` ensures the reminder lands in the original Slack thread with full conversation context
- **Self-cleaning** — the prompt instructs the agent to remove the one-shot entry; worst case if cleanup fails, the entry fires once more next year (specific day/month)
- **OpenCode already has rw crontab access** — no new permissions needed

## Decision Log

| #   | Decision                                              | Rationale                                                                                                                                                                                                                                                   |
| --- | ----------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| D1  | **Reuse cron system, not a new `/schedule` endpoint** | Zero new code for scheduling. The crontab + crond already handles "execute at time X". The EventQueue was designed for short debounce delays (3s–60s), not hour-scale timers — using it for 2h would waste 72K scans and operate outside its design intent. |
| D2  | **Add optional `correlationKey` to `POST /cron`**     | Allows the reminder to resume the original session. Without this, each cron invocation gets a new session (by design). One small schema change, backwards-compatible (field is optional).                                                                   |
| D3  | **`--key` flag on `hey-thor` CLI**                    | Passes the correlation key from crontab to the gateway request. Minimal shell script change.                                                                                                                                                                |
| D4  | **Marker comments (`# ONE-SHOT:<id>`) for cleanup**   | Agent can identify and remove its own one-shot entries. The ID is a short random string generated at write time.                                                                                                                                            |
| D5  | **Self-cleanup via prompt instruction**               | The reminder prompt includes "remove the ONE-SHOT:abc123 entry from /workspace/cron/crontab". The agent knows the file path and has write access. No external cleanup daemon needed.                                                                        |
| D6  | **Specific cron expression (min hour day month `*`)** | Using day-of-month + month makes the entry fire only once per year if cleanup fails. Acceptable failure mode for MVP.                                                                                                                                       |

## Phases

### Phase 1 — `--key` flag on `hey-thor` + optional `correlationKey` on `POST /cron` + inject key into prompt

**Goal**: Allow cron triggers to resume an existing session by passing a correlation key. Ensure every OpenCode session knows its own correlation key.

Steps:

1. Inject correlation key into every prompt in `packages/runner/src/index.ts`:
   - After the pinned memory injection (around line 364), prepend the correlation key to every prompt:
     ```ts
     if (correlationKey) {
       prompt = `[correlation-key: ${correlationKey}]\n\n${prompt}`;
     }
     ```
   - This goes on every prompt (not just the first), so the agent always has it available even after context compression

2. Update `packages/gateway/src/cron.ts`:
   - Add optional `correlationKey` field to `CronRequestSchema`:
     ```ts
     export const CronRequestSchema = z.object({
       prompt: z.string().min(1),
       correlationKey: z.string().optional(),
     });
     ```
   - Update `CronPayload` to include optional `correlationKey`

3. Update `POST /cron` handler in `app.ts`:
   - If `correlationKey` is provided in the request body, use it instead of `deriveCronCorrelationKey()`
   - Otherwise, derive as before (backwards-compatible)

4. Update `docker/cron/hey-thor`:

   ```sh
   #!/bin/sh
   set -e

   KEY=""
   while [ $# -gt 1 ]; do
     case "$1" in
       --key) KEY="$2"; shift 2 ;;
       *) break ;;
     esac
   done

   [ -z "$1" ] && { echo "Usage: hey-thor [--key <correlationKey>] \"<prompt>\""; exit 1; }

   JSON="{\"prompt\":$(printf '%s' "$1" | jq -Rs .)}"
   [ -n "$KEY" ] && JSON="{\"prompt\":$(printf '%s' "$1" | jq -Rs .),\"correlationKey\":$(printf '%s' "$KEY" | jq -Rs .)}"

   exec curl -sf -X POST "${GATEWAY_URL}/cron" \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer ${CRON_SECRET}" \
     -d "$JSON"
   ```

5. Unit tests:
   - `POST /cron` with `correlationKey` → enqueues with that key (not derived)
   - `POST /cron` without `correlationKey` → derives as before
   - `hey-thor --key "slack:C06:123" "prompt"` → sends correlationKey in payload
   - Runner injects `[correlation-key: ...]` into prompt when correlationKey is present

**Exit criteria**:

- Every prompted session sees `[correlation-key: ...]` at the top of its prompt
- `hey-thor --key "slack:C06XXX:1710000000" "test"` triggers a cron event with the provided correlation key
- `hey-thor "test"` still works as before (auto-derived key)
- All existing cron tests still pass
- New tests pass

### Phase 2 — OpenCode instructions for one-shot scheduling

**Goal**: Teach OpenCode how and when to write one-shot crontab entries.

Steps:

1. Add scheduling instructions to OpenCode's pinned memory (`/workspace/memory/ALWAYS.md` or equivalent):

   ```markdown
   ## Scheduling Tasks via Cron

   You can schedule tasks by editing `/workspace/cron/crontab`. There are two patterns:

   ### Recurring jobs (hey-thor)

   When a user asks you to "do X every day", "check Y every 6 hours", "run Z on weekdays", etc., add a recurring cron entry:

   1. Choose the right cron expression for the schedule
   2. Append to `/workspace/cron/crontab`:
   ```

   # <descriptive comment>

   <min> <hour> <dom> <month> <dow> hey-thor "<prompt>"

   ```
   3. Confirm the schedule with the user

   Examples:
   - "check errors every 6 hours" → `0 */6 * * *  hey-thor "Check PostHog for error spikes in the last 6 hours. Post findings to #acme-general on Slack."`
   - "send a standup summary every weekday at 9am" → `0 2 * * 1-5  hey-thor "Generate a standup summary... Post to #acme-general on Slack."`

   Notes:
   - Do NOT use `--key` for recurring jobs — each invocation should get its own session
   - Always include the output destination in the prompt (e.g. "post to #channel on Slack")
   - Be specific about data sources, time windows, and output format in the prompt
   - The crontab uses the container's timezone (UTC)

   ### One-shot reminders (hey-thor --key)

   When a user asks you to "remind me in X" or "do Y in 2 hours", schedule a one-shot entry that resumes the current session:

   1. Calculate the target time from the user's request
   2. Generate a short random ID (e.g. 6 hex chars)
   3. Append to `/workspace/cron/crontab`:
   ```

   # ONE-SHOT:<id>

   <min> <hour> <day> <month> \* hey-thor --key "<your-current-correlation-key>" "<prompt>. After completing this task, remove the lines tagged ONE-SHOT:<id> from /workspace/cron/crontab."

   ```
   4. Confirm the scheduled time with the user

   Notes:
   - Use `--key` with your current correlation key so the reminder lands in the same Slack thread
   - Use specific day + month in the cron expression (not `*`) so it only fires once
   - Always include the cleanup instruction in the prompt
   - Always include the output destination (e.g. "post to #channel on Slack")

   ### Managing cron jobs

   - To list jobs: read `/workspace/cron/crontab`
   - To remove a job: edit the file and remove the relevant lines (comment + cron line)
   - To modify a job: edit the cron line in place
   - Changes take effect within 1 minute (crond re-reads automatically)
   ```

2. Test end-to-end:
   - **Recurring**: ask OpenCode to "check for errors every 6 hours and post to Slack" → verify it writes a correct recurring crontab entry without `--key`
   - **One-shot**: ask OpenCode to "remind me in 1 minute to say hello" → verify it writes the crontab entry with `--key` → verify cron fires → verify response appears in same thread → verify cleanup

**Exit criteria**:

- OpenCode correctly writes recurring crontab entries with `hey-thor` (no `--key`)
- OpenCode correctly writes one-shot crontab entries with `hey-thor --key` and cleanup instructions
- Reminder fires in the same session/Slack thread
- Agent cleans up the one-shot entry after firing
- Agent can list, modify, and remove cron jobs when asked

## Examples

### Recurring job

```
User: check for error spikes every 6 hours and post to #acme-general
```

OpenCode appends to `/workspace/cron/crontab`:

```crontab
# Error spike monitor (every 6 hours)
0 */6 * * *  hey-thor "Check PostHog for error rate spikes in the last 6 hours. If any endpoint shows >20% increase, investigate recent GitHub merges in acme/acme-project for likely causes. Post findings to #acme-general on Slack."
```

OpenCode replies:

```
Done — I've added a cron job that checks for error spikes every 6 hours and posts to #acme-general.
```

### One-shot reminder

User in Slack thread `slack:C06ABC:1710600000`:

```
User: remind me in 2 hours to check if the deployment went through
```

OpenCode appends to `/workspace/cron/crontab`:

```crontab
# ONE-SHOT:f7a3b1
30 17 17 03 *  hey-thor --key "slack:C06ABC:1710600000" "Reminder: the user asked to be reminded to check if the deployment went through. Check the deployment status of the latest release in acme/acme-project. Post findings to this Slack thread. After completing this task, remove the lines tagged ONE-SHOT:f7a3b1 from /workspace/cron/crontab."
```

OpenCode replies:

```
I've scheduled a reminder for 5:30 PM (2 hours from now). I'll check the deployment status and report back in this thread.
```

2 hours later, crond fires → gateway receives with correlation key → runner resumes the session → agent checks deployment and replies in the same Slack thread → agent removes the crontab entry.

## Out of Scope

- **Guaranteed cleanup** — if one-shot cleanup fails, the entry fires once more next year (acceptable)
- **Sub-minute precision** — cron granularity is 1 minute
- **Timezone handling** — crontab uses container timezone (UTC); the agent calculates accordingly
- **Cron expression validation** — trust the agent to write valid expressions; crond silently ignores bad lines
