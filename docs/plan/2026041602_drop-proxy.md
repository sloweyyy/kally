# Drop `@thor/proxy` — Consolidate into `@thor/remote-cli`

**Goal**: Eliminate `packages/proxy` as a separate service. Move all MCP-proxy logic (upstream connections, tool listing/calling, policy enforcement, approval workflow) into `packages/remote-cli` as `/exec/mcp` and `/exec/approval` endpoints. One fewer container, one fewer network hop, consistent API.

## Motivation

- The proxy and remote-cli are both Express HTTP servers with nearly identical patterns (JSON request/response, `{ stdout, stderr, exitCode }` shape, Thor tracing headers, server-side policy).
- The proxy's only unique capability is acting as an MCP client to upstream servers. This is a library concern (`@modelcontextprotocol/sdk`), not an architectural one.
- Merging reduces:
  - 1 Docker container + health check
  - 1 port allocation (3001)
  - Network hops for tool calls
  - Config surface (`proxyPort`, `proxyHost`, `THOR_PROXY_URL`)
- Unifying to `/exec/mcp` pattern makes the API consistent: git, gh, scoutqa, langfuse, metabase, mcp all use `POST /exec/{tool}`.

## Scope

**In scope:**

- Move MCP upstream connection logic into remote-cli
- Move policy classification + approval workflow into remote-cli
- Expose as `POST /exec/mcp` and `POST /exec/approval` (unified API pattern)
- Delete `proxy-cli.ts` (remote-cli.mjs already dispatches to `/exec/{endpoint}`)
- Update gateway approval resolution to hit remote-cli
- Update docker-compose, Dockerfile, runner config
- Delete `packages/proxy`
- Rename `getRepoProxies` → `getRepoUpstreams` in common (in blast radius)
- Add `RESOLVE_SECRET` gating on approval resolution endpoint (security hardening)

**Out of scope:**

- Changing the MCP protocol or approval UX
- Changing the policy model (allow/approve/hidden stays)
- Refactoring remote-cli's existing endpoints (git/gh/scoutqa/langfuse/metabase)

## Current Architecture

```
opencode-cli
  ├─ remote-cli.mjs  → POST remote-cli:3004/exec/{git,gh,scoutqa,langfuse,metabase}
  └─ proxy-cli.mjs   → GET/POST proxy:3001/{upstream}/tools[/call], /approval/*

gateway → POST proxy:3001/{upstream}/approval/{id}/resolve
```

## Target Architecture

```
opencode-cli
  └─ remote-cli.mjs  → POST remote-cli:3004/exec/{git,gh,scoutqa,langfuse,metabase,mcp,approval}

gateway → POST remote-cli:3004/exec/mcp  {args: ["resolve", id, decision, reviewer]}
                                         + x-thor-resolve-secret header
```

### API Design

MCP endpoint (`POST /exec/mcp`):

```
Body: { args: string[], cwd: string, directory: string }

cwd = current shell working directory
directory = trusted OpenCode session directory used for MCP authz

args = []                                          → list upstreams
args = ["<upstream>"]                              → list tools
args = ["<upstream>", "<tool>", "--help"]           → tool schema
args = ["<upstream>", "<tool>", "<json-args>"]      → call tool

Response: { stdout, stderr, exitCode }
```

Approval endpoint (`POST /exec/approval`):

```
Body: { args: string[] }

args = ["status", "<id>"]   → check approval status
args = ["list"]              → list pending approvals

Response: { stdout, stderr, exitCode }
```

Resolve via secret-gated `/exec/mcp`:

```
POST /exec/mcp
Headers: { x-thor-resolve-secret: string }  ← must match RESOLVE_SECRET env var
Body: { args: ["resolve", "<actionId>", "<decision>", "<reviewer>", "<reason>?"] }

WITHOUT secret header: → { stderr: "Unknown subcommand: resolve", exitCode: 1 }
WITH secret header:    → lookup approval by actionId → execute stored tool+args → return result

Response: { stdout, stderr, exitCode }
```

Same endpoint, secret as auth. The agent's `mcp` CLI wrapper never sends the secret header, so `mcp resolve ...` from the agent always fails. Gateway sends the secret, so its resolve calls succeed. No new endpoints needed.

### Phase 1 — Move MCP proxy logic into remote-cli

**Files to create/modify:**

| File                                        | Action          | Notes                                                                                                                      |

| `packages/remote-cli/src/index.ts`          | Edit            | Add `/exec/mcp` and `/exec/approval` routes, add startup logic for eager upstream connect, update health endpoint          |

**No tsup config change needed.** Root `tsup.config.ts` already handles this: `noExternal` only inlines `@thor/*` workspace packages. The MCP SDK stays in `node_modules` and is not bundled.

**Startup sequence:** CLI endpoints (`/exec/git`, `/exec/gh`, etc.) available at T+0. MCP upstreams connect asynchronously in the background. Health endpoint always returns 200 if Express is up. Response body includes MCP upstream info for debugging. Docker compose healthcheck uses this simple alive check.

**MCP logger:** Use `createLogger("mcp")` for MCP-related log events, separate from the existing `createLogger("remote-cli")`.

**CWD validation for MCP:** The existing `validateCwd()` from `policy.ts` validates paths. MCP endpoints should use this same validation since `cwd` is now a body field instead of an `x-thor-directory` header. Note: `/exec/approval` does NOT require cwd (approvals are not repo-scoped).

**Approval store path:** Use absolute path `/workspace/data/approvals` (not relative). The proxy uses `data/approvals` relative to its WORKDIR (`/workspace`). Remote-cli has a different WORKDIR. Use an absolute path to avoid silent data loss.

**Exit criteria:**

- `GET /health` returns 200 with MCP upstream info in body
- `POST /exec/mcp` with `{args: [], cwd: "...", directory: "..."}` returns upstream list
- `POST /exec/mcp` with `{args: ["slack", "postMessage", '{"...":"..."}'], cwd: "...", directory: "..."}` calls tool
- `POST /exec/approval` with `{args: ["status", "uuid"]}` returns approval
- `POST /exec/mcp` with `{args: ["resolve", ...]}` + secret header resolves approval
- `POST /exec/mcp` with `{args: ["resolve", ...]}` WITHOUT secret returns error
- All existing remote-cli endpoints (git/gh/scoutqa/langfuse/metabase) unaffected
- Proxy tests adapted and passing against remote-cli
