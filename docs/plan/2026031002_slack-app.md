# Plan — 2026031002_slack-app

> Add a Slack app service that turns Slack thread mentions into runner triggers and posts Thor's results back into the same thread.

## Decision Log

| #   | Decision                                                | Rationale                                                                      |
| --- | ------------------------------------------------------- | ------------------------------------------------------------------------------ |
| D1  | Add a dedicated `packages/gateway` service              | Keeps Slack-specific auth, webhook handling, and retries out of runner.        |
| D2  | Use Slack Events API `app_mention` as MVP input         | Matches the MVP trigger model and avoids slash-command specific plumbing.      |
| D3  | Verify signatures with Node `crypto`                    | No Slack SDK needed for Phase 1; keeps dependencies small and explicit.        |
| D4  | Acknowledge Slack before running the agent              | Slack events must be ACKed quickly; runner work can finish asynchronously.     |
| D5  | Reuse runner session continuity via thread key          | `slack:thread:<thread_ts>` aligns with the existing correlation model.         |
| D6  | Post replies with Slack Web API `chat.postMessage`      | Produces a complete thread loop without adding MCP Slack tooling yet.          |
| D7  | Skip OAuth/install flow for now                         | Workspace-scoped bot token + signing secret are enough for local MVP.          |
| D8  | Add a root Vitest alias for `@thor/common`              | Lets workspace package tests run from source without requiring prebuilds.      |
| D9  | Expose manifest-configured Slack routes early           | Keeps deployed request URLs valid even before OAuth/interactivity logic lands. |
| D10 | ~~Async queue for thread replies deferred to Phase 4~~  | Superseded by D13 — directory-based queue handles all events.                  |
| D11 | Rename `slack-app` → `gateway`                          | Service will accept webhooks from multiple sources (Slack, Jira, etc.).        |
| D12 | Thread replies forward if session exists                | Gateway calls `GET /sessions?correlationKey=...` before triggering runner.     |
| D13 | Directory-based event queue replaces in-memory debounce | Eliminates concurrency bugs by design; per-key serial processing.              |
| D14 | Batch all queued events per key (no dropping)           | Agent sees every message; batched into a single prompt for efficiency.         |
| D15 | Fire-and-forget runner triggers (future)                | Gateway should not block on runner response; runner posts replies directly.    |

## Architecture

```text
Slack Events API
    |
    v
packages/gateway (/slack/events)
    | verify signature + normalize mention
    | enqueue event as JSON file
    v
Directory-based event queue (data/queue/)
    | fs.watch + settle timer (100ms)
    | group by correlationKey, batch per key
    | per-key serial processing
    v
Queue handler
    | combine prompts from batched events
    | POST /trigger { prompt, correlationKey }
    v
packages/runner
    | abort-if-busy + wait-for-idle on resumed sessions
    v
OpenCode -> proxy -> MCP upstreams
    |
    v
packages/gateway -> Slack chat.postMessage (same thread)
```

## Phases

### Phase 1 — Slack Mention Vertical Slice ✅

**Goal**: A Slack `app_mention` event creates or resumes a runner session and posts Thor's response back into the same Slack thread.

Steps:

1. ✅ Add `packages/gateway/` TypeScript service with Express server and `/health`
2. ✅ Add `POST /slack/events` with:
   - Slack signature verification (`X-Slack-Signature`, `X-Slack-Request-Timestamp`)
   - `url_verification` support
   - `app_mention` event handling
3. ✅ Normalize mention payload into:
   - `prompt`: event text with bot mention removed
   - `correlationKey`: `slack:thread:<thread_ts || ts>`
4. ✅ Forward to runner `POST /trigger` (via directory-based event queue)
5. ✅ Post the runner result back to Slack with `chat.postMessage` in the originating thread (skipped on empty response or error)
6. ✅ Add unit tests for signature verification and prompt normalization
7. ✅ Wire the service into `docker-compose.yml` and workspace scripts/docs as needed
8. ✅ Expose manifest-aligned placeholders for `/slack/interactivity` and `/slack/redirect`
9. ✅ Directory-based event queue (`queue.ts`) with fs.watch, settle timer, per-key serial processing, and batch-all-events semantics (D13, D14)
10. ✅ Queue handler combines prompts from batched events into a single runner trigger

**Test** (45 tests across 5 files):

- Unit tests for signature verification and prompt extraction (`slack.test.ts`)
- Event queue unit tests: batching, independent keys, in-flight arrivals, file cleanup, corrupt files, error recovery (`queue.test.ts`)
- Integration-style tests with mocked `fetch`: runner forwarding, Slack reply posting, rapid message coalescing, sequential processing (`app.test.ts`)

### Phase 2 — Delivery Hardening

**Goal**: Make Slack delivery safe against retries, malformed payloads, and noisy events.

Steps:

1. Ignore Slack retries using `X-Slack-Retry-Num` and event IDs
2. ✅ Reject unsupported event types cleanly (returns `200 { ok, ignored }`)
3. Add configurable timeout/retry handling for runner and Slack API calls
4. ✅ Add structured logging for inbound events and outbound replies (`createLogger("gateway")`)
5. Add failure-path Slack reply formatting (currently errors are logged only, not replied)

**Exit criteria**:

- Duplicate Slack deliveries do not trigger duplicate runner calls
- Timeouts and Slack API failures are logged with enough context to debug

### Phase 3 — Slack Approval + Notification Surface

**Goal**: Reuse the Slack app credentials/channel rules for outbound system notifications and approval prompts.

Steps:

1. Define channel allowlist config for outbound posting
2. Add reusable Slack client module shared by trigger + notification flows
3. Expose approval notification hooks for the proxy
4. Document operational setup for Slack app scopes and event subscriptions

**Exit criteria**:

- Proxy can send approval notifications through the shared Slack posting layer
- Channel restrictions are enforced centrally

### Phase 4 — Async Thread Reply Processing ✅

> **Partially obsolete** — see `2026041301_slack-drop-unengaged.md`. Non-mention messages are now only forwarded if Thor has already replied in the thread (engaged). Unengaged threads are dropped instead of queued with a 60s delay.

**Goal**: Allow non-mention thread replies to trigger the agent without conflicting with in-flight prompts.

Steps:

1. Subscribe to `message.channels` / `message.groups` in Slack app config (app manifest config — outside codebase)
2. ✅ ~~Gateway queues non-mention thread messages per correlation key~~ — now only if thread is engaged
3. ✅ Runner exposes session busy/idle status (`session.status()`) and abort (`session.abort()`)
4. ✅ ~~Gateway checks session existence via `GET /sessions?correlationKey=...` before forwarding thread replies~~ — replaced by `hasSlackReply()` check
5. ✅ On busy: runner aborts in-flight prompt, waits for `session.idle`, then re-prompts (`runner/src/index.ts`)
6. ✅ Filter out bot's own messages (`bot_id`) and message subtypes to prevent loops (`app.ts:169`)

**Exit criteria**:

- ✅ Thread replies without mentions resume the existing session — only in engaged threads
- ✅ Concurrent messages on the same thread do not cause runner errors (per-key serial queue + abort-if-busy)
- ✅ Bot messages are not re-processed

### Phase 5 — Fire-and-Forget Runner Triggers

**Goal**: Gateway should not block on runner responses. Runner posts replies directly to Slack.

**Context**: Currently the gateway blocks on the runner HTTP response (which can take minutes) just to relay the reply to Slack. This ties up the queue's per-key serial processing and limits throughput.

Steps:

1. Runner accepts a callback/reply context in the trigger payload (channel, thread_ts, bot token or callback URL)
2. Runner `POST /trigger` returns `202 Accepted` immediately
3. Runner posts results directly to Slack (or calls back to gateway) after processing
4. Gateway queue becomes fire-and-forget — enqueue, trigger, done (~100ms per event)
5. Abort-if-busy logic stays in runner (gateway no longer waits for completion)

**Exit criteria**:

- Gateway trigger calls return in <500ms regardless of prompt duration
- Runner posts threaded Slack replies independently
- Per-key queue throughput is no longer bounded by prompt processing time
