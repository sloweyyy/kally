# Plan: MCP Tool CLI — stateless tool access for OpenCode

**Date**: 2026-04-02
**Status**: Draft

## Goal

Replace OpenCode's persistent MCP connections to the proxy with a stateless CLI that lists and calls MCP tools over plain HTTP. The proxy can restart freely without breaking the agent's tool access.

## Why

OpenCode holds SSE/Streamable HTTP connections from startup. When the proxy restarts (config reload, deploy, crash), those connections break. The agent loses MCP sessions and must be restarted. A stateless CLI avoids this — each invocation is one HTTP round-trip, no session to break.

This follows the same pattern already proven with `git`/`gh` — wrapper scripts in the OpenCode container that call an HTTP service. The MCP tool CLI extends `remote-cli.mjs` to cover MCP tools.

## CLI Design

**Name**: `mcp` — short, accurate, no conflicts inside the controlled container PATH.

### Usage

Upstream and tool names are the subcommands — no `list`/`call` ceremony:

```bash
mcp                                              # list available upstreams (from config.json)
mcp slack                                        # list tools on the slack upstream
mcp slack postMessage '{"channel":"C123","text":"hello"}'  # call a tool (JSON arg → JSON response)
```

- `mcp` or `mcp --help` or `mcp -h` → print available upstreams
- `mcp <upstream>` or `mcp <upstream> --help` → print tool list for that upstream
- `mcp <upstream> <tool> --help` → print tool description and input schema (from proxy)
- `mcp <upstream> <tool> '<json>'` → call the tool, single JSON string argument

Tool names are passed through as-is (underscores, camelCase, etc.).

### Approval CLI

Separate CLI for the approval workflow — different concern from tool discovery/execution.

```bash
approval status <action-id>     # check approval status (pending/approved/rejected + result)
approval list                   # list pending approvals (useful for debugging)
```

- Same `remote-cli.mjs` pattern — `docker/opencode/bin/approval` shell wrapper
- Hits proxy endpoints: `GET /approval/:id` and `GET /approvals`
- The proxy's synthetic `check_approval_status` MCP tool is removed — no MCP session clients remain

### Output format

- Raw MCP JSON response (as returned by the proxy)
- No pretty-printing or transformation — keeps it simple, parseable by the agent

### Per-repo upstream scoping

Currently, each repo's `opencode.json` controls which MCP servers it connects to. With the CLI, this moves into `config.json` — each repo declares which proxies it can access:

```json
{
  "repos": {
    "acme-app": {
      "channels": ["C123"],
      "proxies": ["slack", "atlassian", "grafana"]
    },
    "data-pipeline": {
      "channels": ["C456"],
      "proxies": ["slack", "posthog"]
    }
  },
  "proxies": {
    "slack": { ... },
    "atlassian": { ... },
    "posthog": { ... },
    "grafana": { ... }
  }
}
```

**Enforcement**: The CLI sends `cwd` with every request (same as `git`/`gh` wrappers). The proxy extracts the repo name from the path (e.g. `/workspace/repos/acme-app` → `acme-app`) and filters upstreams to only those listed in that repo's `proxies` array. A repo without a `proxies` field gets no upstreams.

No extra env var or header needed — repo identity is derived from the working directory, which is already sent by `remote-cli.mjs`.

This replaces the per-repo `opencode.json` MCP block with a single centralized config — upstream definitions stay DRY, and the proxy enforces access server-side.

## Architecture

```
OpenCode agent (cwd=/workspace/repos/acme-app)
  │
  ├─ git push ...                         → remote-cli.mjs git      → POST http://remote-cli:3004/exec/git    {args, cwd}
  ├─ gh pr create ...                     → remote-cli.mjs gh       → POST http://remote-cli:3004/exec/gh     {args, cwd}
  ├─ mcp                                  → remote-cli.mjs mcp      → POST http://proxy:3001/tools            {cwd}
  ├─ mcp slack                            → remote-cli.mjs mcp      → POST http://proxy:3001/slack/tools      {cwd}
  ├─ mcp slack postMessage '{"...":""}'  → remote-cli.mjs mcp      → POST http://proxy:3001/slack/tools/call  {name, arguments, cwd}
  ├─ approval status <id>                 → remote-cli.mjs approval → GET  http://proxy:3001/approval/<id>
  └─ approval list                        → remote-cli.mjs approval → GET  http://proxy:3001/approvals
```

The `mcp` wrapper reuses the same `remote-cli.mjs` pattern but targets the proxy instead of the remote-cli service. Each invocation is one HTTP request — no session, no SSE, no streaming transport.

All requests include `cwd` in the JSON body. The proxy extracts the repo name from the path and validates access before proceeding.

### Config schema change

Add optional `proxies` array to `RepoConfigSchema`:

```typescript
const RepoConfigSchema = z.object({
  channels: z.array(z.string()).optional(),
  proxies: z.array(z.string()).optional(), // new — allowed upstream names
});
```

Validation: each entry in `proxies` must reference a key in the top-level `proxies` map.

### Proxy changes needed

The proxy currently requires MCP session initialization (`initialize` → `initialized` handshake) before accepting `tools/list` or `tools/call`. For stateless CLI access, we need **sessionless HTTP endpoints**:

**Tool endpoints:**

- `POST /tools` — returns upstreams available to the repo (filtered by `cwd` in body)
- `POST /:upstream/tools` — returns tool list (no MCP session required, validates repo access)
- `POST /:upstream/tools/call` — executes a tool call (no MCP session required, validates repo access)

**Approval endpoints:**

- `GET /approval/:id` — check approval status by action ID (searches across all upstream stores)
- `GET /approvals` — list pending approvals (optional, for debugging)

**Removals:**

- Remove the `check_approval_status` synthetic tool from MCP session handlers — no MCP session clients remain after migration
- Remove MCP session endpoints (`POST/GET/DELETE /:upstream` for MCP transport) — replaced by the stateless endpoints above

All tool endpoints read `cwd` from request body to scope access. If the repo doesn't have the requested upstream in its `proxies` list, return 403.

### OpenCode config change

Remove all `mcp` entries from `opencode.json`:

```json
{
  "model": "opencode/big-pickle",
  "permission": "allow",
  "mcp": {}
}
```

Tools become available via agent instructions (system prompt) that tell the agent to use the `mcp` CLI.

## Phases

### Phase 1 — Config schema + sessionless proxy endpoints

Extend the config schema and add plain HTTP endpoints to the proxy.

1. Add `proxies` field to `RepoConfigSchema` in `workspace-config.ts`
   - Validate entries reference keys in top-level `proxies` map
   - Add unit tests for the new validation
2. Add `POST /tools` endpoint (list upstreams for a repo)
   - Reads `cwd` from request body, extracts repo name from path
   - Returns `{ upstreams: [{ name, description, toolCount }...] }` filtered by repo's `proxies` list
3. Add `POST /:upstream/tools` endpoint
   - Validates repo access via `cwd` in request body (403 if not allowed)
   - Reads workspace config (live reload)
   - Connects to upstream if not already connected
   - Runs tool classification (allow/approve/hidden filtering)
   - Returns `{ tools: [...] }` JSON (including description per tool)
4. Add `POST /:upstream/tools/call` endpoint
   - Validates repo access via `cwd` in request body (403 if not allowed)
   - Request body: `{ name: string, arguments: object }`
   - Runs same policy logic as MCP `CallToolRequest` handler (allow/approve/hidden)
   - For `allow`: forwards to upstream, returns result
   - For `approve`: creates pending approval, returns action ID
   - For `hidden`: returns error
   - Returns MCP `CallToolResult` JSON
5. Add `GET /approval/:id` endpoint
   - Searches across all upstream approval stores by action ID
   - Returns approval status (pending/approved/rejected) + result if approved
6. Add `GET /approvals` endpoint (optional, for debugging)
   - Returns list of pending approvals across all upstreams
7. Add `tools`, `approval`, `approvals` to `RESERVED_PROXY_NAMES` to prevent route collisions
8. Register all new static-path routes (`/tools`, `/approvals`, `/approval/:id`) before `/:upstream` wildcard routes
9. Unit tests for the new endpoints
   - Note: MCP session endpoints and `check_approval_status` stay in Phase 1 (removed in Phase 3 alongside OpenCode MCP disconnection)

### Phase 4 — Error handling and edge cases

Make the CLI helpful when things go wrong. Fuzzy matching, auto-suggestions, and clear error messages.

#### 4a. Fuzzy matching and suggestions

When the agent makes a typo or guesses wrong, the CLI should guide it to the right answer:

1. **Unknown upstream** — `mcp slck`:
   - Fuzzy match against configured upstreams (Levenshtein or substring)
   - Output: `Unknown upstream "slck". Did you mean "slack"? Available upstreams: slack, atlassian, posthog, grafana`
   - Exit 1

2. **Unknown tool** — `mcp slack post_msg`:
   - Fuzzy match against that upstream's tool list
   - Output: `Unknown tool "post_msg" on upstream "slack". Did you mean "post_message"? Available tools: post_message, read_thread, get_channel_history, get_slack_file`
   - Exit 1

3. **Tool call schema error** — upstream rejects the arguments:
   - The upstream MCP server returns the error in CallToolResult (no client-side validation)
   - CLI auto-appends the tool's input schema to the error output so the agent can self-correct without a separate `--help` call
   - Output: raw CallToolResult JSON with error, followed by `\n[hint] Input schema for "post_message":\n<schema JSON>`

#### 4b. Approval flow

The approval flow must produce the same JSON signals the runner already parses (`{"type":"approval_required","actionId":"...","proxyName":"...","tool":"..."}`).

1. **Approve-gated tool call** — `mcp atlassian createJiraIssue '{...}'`:
   - Proxy returns CallToolResult with the `approval_required` JSON content item (existing format)
   - The response text says `Run: approval status <id>` (instead of referencing the removed `check_approval_status` tool)
   - CLI outputs raw CallToolResult to stdout — runner parses the `approval_required` signal as before
   - No change to runner parsing logic needed

2. **Check approval status** — `approval status abc123`:
   - Separate CLI, not part of `mcp`
   - Hits `GET /approval/:id` on the proxy
   - Proxy looks up the action ID across all upstream approval stores
   - Returns approval status JSON (pending/approved/rejected + result if approved)

#### 4c. Connection and runtime errors

4. **Proxy unreachable** — `mcp slack post_message '{...}'` but proxy is down:
   - stderr: `Failed to reach proxy at http://proxy:3001: connection refused`
   - Exit 1

5. **Upstream unreachable** — proxy is up but upstream MCP server is down:
   - Proxy returns 502 with error message
   - CLI outputs: `Upstream "slack" is unreachable: connection refused to http://slack-mcp:3003/mcp`
   - Exit 1

6. **Upstream timeout** — tool call takes too long:
   - CLI timeout: 120s (same as remote-cli defaults)
   - stderr: `Timeout calling "post_message" on upstream "slack" (120s)`
   - Exit 1

7. **Bad cwd** — `cwd` is not under `/workspace/repos/`:
   - stderr: `Cannot determine repo from cwd: /tmp. Expected /workspace/repos/<repo>`
   - Exit 1

8. **Repo not in config** — cwd resolves to a repo not in config.json:
   - stderr: `Repo "unknown-repo" not found in config. Available repos: acme-app, data-pipeline`
   - Exit 1

9. **Repo has no `proxies` field** — repo exists but no upstreams configured:
   - `mcp` returns: `No upstreams configured for repo "acme-app".`
   - Exit 0 (not an error, just empty)

10. **Config changed mid-session** — tool was available, now removed:
    - Tool call fails with "Unknown tool" error + fuzzy suggestion (same as #2)
    - Agent can re-run `mcp` / `mcp <upstream>` to refresh its knowledge

**Exit criteria:**

- [ ] `mcp slck` suggests "slack" via fuzzy match
- [ ] `mcp slack post_msg` suggests "post_message" via fuzzy match
- [ ] Schema errors auto-include the tool's input schema as a hint
- [ ] `approval status <id>` returns approval status from proxy
- [ ] Approval flow produces same JSON signals runner already parses
- [ ] All error cases output helpful messages to stderr with exit 1
- [ ] `pnpm test` passes
