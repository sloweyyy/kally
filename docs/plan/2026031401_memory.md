# Agent Memory — 2026-03-13-01

> Give Thor durable, cross-session memory so it accumulates knowledge over time — not just conversation history within a single OpenCode session, but structured facts and context that survive session loss, compaction, and container restarts.

## Problem

Today Thor has **session continuity** (correlation key → OpenCode session ID) and **working notes** (markdown audit trail per session). But:

1. **Notes are write-only** — never read back as context when resuming. If an OpenCode session is lost (compacted, deleted, container restart), all accumulated knowledge is gone.
2. **No structured memory** — the agent can't persist discovered facts (e.g., "ACME-456 is blocked on API approval") in a way that's retrievable by future sessions or different correlation keys.
3. **No cross-session knowledge** — session A about a Slack thread can't access what session B learned about a related Jira issue.

## Decision Log

| #   | Decision                                            | Rationale                                                                                                            |
| --- | --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| D1  | **Plain markdown, no YAML frontmatter**             | Keep it simple. The agent writes natural markdown. No parsing library needed.                                        |
| D2  | **No session index file**                           | Existing notes-based lookup works. Directory scan is fast enough for PoC.                                            |
| D3  | **Memory directory, agent-managed via file tools**  | `/workspace/memory/` holds markdown files. OpenCode reads/writes directly — no custom MCP tools or utilities needed. |
| D4  | **Resumption context on stale sessions only**       | Only inject notes context when a session is lost. Normal resumes have full history via OpenCode.                     |
| D5  | **LLM decides what to remember and what to recall** | No programmatic memory injection. System prompt tells the agent to check memory at session start.                    |
| D6  | **Keep notes append-only, memory mutable**          | Notes remain an audit trail. Memory files are mutable knowledge the agent curates.                                   |
| D7  | **No new dependencies**                             | No `yaml` parser. No new `@thor/common` modules for memory. Just system prompt + directory conventions.              |

## Architecture

```
/workspace/
├── memory/                         ← agent-writable, agent-readable (separate mount)
│   └── *.md                        ← plain markdown, any structure the agent chooses
└── worklog/
    ├── 2026-03-12/
    │   └── notes/
    │       └── slack-thread-123.md  ← frozen session notes
    └── 2026-03-13/
        └── notes/
            └── slack-thread-123.md  ← today's continuation
```

## Phases

### Phase 1 — Resumption Hint on Stale Sessions

**Goal**: When a session is lost (stale session ID → new session created), point the agent to the previous notes file so it can recover context on its own.

This is the only phase that requires code changes. It's a runner concern — the agent can't discover the old notes path itself because the session is already gone.

Steps:

1. Add `getNotesLineCount(filePath)` in `@thor/common/notes.ts`:
   - Count lines in the notes file so the agent can gauge size before reading
2. In the runner, when a session is **stale** (candidate session ID fails verification → new session created):
   - Get the previous notes path via `findNotesFile(correlationKey)` and its line count
   - Prepend a hint to the prompt:

     ```
     [Previous session was lost. Your notes from the prior session are at: worklog/2026-03-12/notes/slack-thread-123.md (47 lines) — read it if you need context.]

     {prompt}
     ```

3. When a session is successfully **resumed** (same session ID), do NOT add hint (OpenCode already has full history)
4. Add a `previousNotesPath` field to the NDJSON `start` event so the gateway/caller knows context was referenced

**Exit criteria**:

- Stale session replacement prepends a hint with the notes path and line count
- Normal session resumption does not add a hint
- The agent can read the referenced file to recover prior context

---

### Phase 2 — Agent-Writable Memory (via system prompt)

**Goal**: Give the agent a persistent memory directory it can read and write during runs. No code changes — just system prompt instructions and directory conventions.

Steps:

1. Create `/workspace/memory/` directory (via bind mount, auto-created on first write)
2. Add memory instructions to the agent's system prompt (in `AGENTS.md` or OpenCode config):
   - **Directory**: `/workspace/memory/` — plain markdown files
   - **The agent decides**: file names, structure, what to remember, how to organize
   - **When to write**: after discovering durable facts useful for future sessions
   - **When to read**: at session start, check `/workspace/memory/` for relevant context (use `Glob` + `Read` or `Grep`)
   - **Updates**: use `Edit` to update existing memory files (mutable, unlike notes)
3. Add a brief `worklog/memory/README.md` with conventions (so the agent can self-reference)

**Exit criteria**:

- Agent can create memory files during a run
- Agent checks memory at session start when instructed via system prompt
- A different session (different correlation key) can read memory written by another session

---

### Phase 3 — Pinned Memory (`ALWAYS.md`)

**Goal**: A special file `/workspace/memory/ALWAYS.md` whose contents are automatically injected into the first prompt of every new or stale session.

Steps:

1. Runner reads `ALWAYS.md` on new/stale sessions (not resumed — OpenCode has full history)
2. Prepend contents as `[Pinned memory — important context from prior sessions]` block
3. System prompt tells the agent about `ALWAYS.md`: what to put there, to keep it short
4. Configurable path via `PINNED_MEMORY_PATH` env var

**Exit criteria**:

- New sessions get `ALWAYS.md` content prepended to the prompt
- Resumed sessions do not get it (already in history)
- Agent can update `ALWAYS.md` during a run
- System prompt documents the convention
