# Plan — 2026031003_slack-mcp

> Add Slack MCP access for OpenCode so Thor can inspect Slack context on demand, while keeping deterministic Slack ingress and thread replies in `packages/gateway`.

## Context

Current state:

- `packages/gateway` accepts Slack Events API webhooks, derives `slack:thread:<ts>` correlation keys, and fires a trigger to the runner (fire-and-forget). The agent replies to Slack on its own via the `post_message` MCP tool through the Slack proxy.
- OpenCode talks to a single `proxy` container that runs one proxy instance per upstream on separate ports (3010 atlassian, 3011 posthog, 3012 slack). Each instance fronts a single MCP server and exposes only its curated allow-list of tools.
- The active Slack app plan explicitly leaves the Slack MCP tool surface out of scope.

New information from current vendor docs:

- Slack now offers an official hosted MCP server at `https://mcp.slack.com/mcp`.
- Slack's hosted MCP docs describe OAuth for Slack **user tokens** for MCP clients, and list required scopes as "scopes needed on user token for different tools", including `chat:write`.
- Slack's broader OAuth docs still support bot tokens for apps, and Slack recommends bot tokens for apps acting independently of a user's identity.
- OpenCode supports remote MCP servers with OAuth config (`clientId`, `clientSecret`, `callbackPort`) and per-agent tool restrictions.

## Options

### Option A — Custom internal Slack MCP server backed by bot token

**Shape**

- Add a small internal MCP server backed by the Slack bot token (`xoxb`)
- Expose only the tools we actually want Thor to have
- Run it either directly in OpenCode or behind `packages/proxy`

**Pros**

- Uses the bot identity we actually want
- Tool behavior is deterministic and scoped to our app's channel membership and bot scopes
- Fits the current proxy model with static credentials
- Lets us restrict the tool surface to safe operations like "read thread", "list channel history", and "post message"
- Keeps posting identity consistent with `packages/gateway`

**Cons**

- We own the server implementation and maintenance
- Bot-token visibility is narrower than user-token search for channels the bot is not in
- We need to define and test each tool ourselves

### Option B — Official hosted Slack MCP direct to OpenCode

**Shape**

- Add a direct `slack` remote MCP entry to `docker/opencode/opencode.docker.json`
- Authenticate it with Slack OAuth
- Use it for broader Slack discovery where user-scoped access is acceptable

**Pros**

- Slack maintains the tool surface
- Better fit for broad search/discovery across Slack
- No need to invent higher-level search tools ourselves

**Cons**

- Posting is user-token-oriented per Slack's MCP docs, so it does not meet the "send as bot" requirement cleanly
- Bypasses the current proxy unless we add OAuth support there
- Requires interactive OAuth bootstrap and token persistence for the OpenCode container

### Option C — Official hosted Slack MCP behind `packages/proxy`

**Shape**

- Teach the proxy to connect to OAuth-backed remote MCP servers
- Route Slack MCP through the same prefixed-tool + policy-validation path as Atlassian/PostHog

**Pros**

- Preserves the intended architecture for policy and audit
- Lets us block or allow Slack tools explicitly
- Keeps tool discovery and logging uniform across integrations

**Cons**

- Still does not solve the bot-vs-user identity mismatch in hosted Slack MCP posting
- Requires OAuth token bootstrap/storage/refresh design inside the proxy
- Slower to ship than the bot-token need in front of us

## Recommendation

Use **Option A now** and keep the system reply path separate:

- Slack Events API ingress and default thread replies remain in `packages/gateway`
- Add a small internal Slack MCP surface backed by the existing bot token model
- Start with a narrow toolset: read thread/channel context and post bot-authored messages
- Keep the official hosted Slack MCP as an optional later add-on for broader read/search use cases

This matches the actual requirement better:

- `gateway` handles deterministic system behavior
- Internal Slack MCP handles discretionary agent behavior using the same bot identity

If later we want richer Slack discovery than a bot token can provide, we can add the official hosted Slack MCP alongside this or behind a future OAuth-capable proxy.

## Hosted MCP Capability Inventory

Slack's public docs describe the hosted server by capability, rate-limit bucket, and required scopes. They do **not** publish the exact MCP JSON-RPC tool names in the docs I could find. So the mirror target below is the documented functional surface, not Slack's internal RPC identifiers.

Documented hosted capabilities:

| Hosted capability in Slack docs | Public evidence             | Mirror priority | Proposed internal tool        |
| ------------------------------- | --------------------------- | --------------- | ----------------------------- |
| Search messages and files       | Overview + rate-limit table | P2              | `slack_search_messages_files` |
| Search users                    | Overview + rate-limit table | P2              | `slack_search_users`          |
| Search channels                 | Overview + rate-limit table | P2              | `slack_search_channels`       |
| Send message                    | Overview + rate-limit table | P1              | `slack_post_message`          |
| Draft message / preview message | Overview only               | P3              | `slack_draft_message`         |
| Read channel history            | Overview + rate-limit table | P2              | `slack_get_channel_history`   |
| Read thread replies             | Overview + rate-limit table | P1              | `slack_read_thread`           |
| Create canvas                   | Overview + rate-limit table | P4              | `slack_create_canvas`         |
| Update canvas                   | Overview + rate-limit table | P4              | `slack_update_canvas`         |
| Read canvas as markdown         | Overview + rate-limit table | P4              | `slack_read_canvas`           |
| Read user profile/email         | Overview + rate-limit table | P3              | `slack_get_user_profile`      |

Notes:

- `Draft message` appears in the feature overview, but it is absent from the public rate-limit and scope tables. I infer this is either a lightweight helper capability or bundled differently than server-side Slack API actions.
- The docs group "search messages and files" together in the rate-limit table but separate file search in the scope table. I would still mirror this as one tool first unless we find a strong need to split it.
- Exact Slack-hosted tool names remain unknown until we can run an authenticated `tools/list` against `https://mcp.slack.com/mcp`.

## Mirror Recommendation

Mirror the hosted surface in layers instead of cloning everything at once:

### Phase 1 mirror set

- `slack_post_message`
- `slack_read_thread`
- `slack_get_channel_history`

This covers the bot-token use cases we already have or are about to need.

### Phase 2 mirror set

- `slack_search_messages_files`
- `slack_search_users`
- `slack_search_channels`
- `slack_get_user_profile`

This matches the hosted server's discovery value as far as bot-token visibility allows.

### Phase 3 mirror set

- `slack_draft_message`
- `slack_create_canvas`
- `slack_update_canvas`
- `slack_read_canvas`

These are useful, but they are not on the critical path for Slack thread operations and approvals.

## Proposed Architecture

```text
Slack Events API
    |
    v
packages/gateway (/slack/events)
    | verify signature, enqueue, batch
    | POST /trigger { prompt, correlationKey }  (fire-and-forget)
    v
packages/runner
    |
    v
OpenCode
    |--- remote MCP --> packages/proxy --> Atlassian / PostHog
    |
    \--- remote MCP --> packages/proxy --> packages/slack-mcp (bot token)
                                              |
                                              v
                                         Slack Web API
                                         (post_message, read_thread, get_channel_history)
```

Notes:

- Gateway enriches the prompt with Slack context (channel, thread_ts) so the agent knows where to reply.
- The agent replies to Slack on its own via `post_message` through the Slack proxy — gateway does not post replies.
- Gateway still handles signature verification, event queuing/batching, eyes reaction, and session-gated thread forwarding.

## Phases

### Phase 1 — Bot-Token Slack MCP Vertical Slice

**Goal**: OpenCode can call a minimal internal Slack MCP server to read Slack thread context and post messages as the bot.

Steps:

1. ✅ Create `packages/slack-mcp` as a standalone MCP server using `@modelcontextprotocol/sdk`
2. ✅ Use Streamable HTTP transport (same pattern as `packages/proxy`)
3. ✅ Authenticate to Slack Web API with bot token from `SLACK_BOT_TOKEN` env var
4. ✅ Implement Phase 1 tool set:
   - `post_message` — post a message to a channel or thread
   - `read_thread` — read all replies in a thread
   - `get_channel_history` — read recent messages from a channel
5. ✅ Add as upstream in `packages/proxy/proxy.slack.json` (`"slack": { "url": "http://slack-mcp:3003/mcp" }`)
6. ✅ Add policy rules: allow `post_message`, `read_thread`, `get_channel_history`; block `*` catch-all already covers rest
7. ✅ Add `slack-mcp` service to `docker-compose.yml` on port 3003
8. ✅ Add `SLACK_BOT_TOKEN` to `.env.example` and docker-compose env

**Exit criteria**:

- OpenCode can call `post_message`, `read_thread`, `get_channel_history` through the Slack proxy
- Thor can read an existing thread and post a bot-authored message via MCP
- Policy rules restrict Slack tool access through the proxy

### Phase 2 — Broader Bot-Scoped Slack Context

**Goal**: Expand the internal Slack MCP surface without drifting into an oversized Slack wrapper.

Steps:

1. Preserve Slack attachment and file/photo metadata in read APIs while keeping `post_message` text-only
2. Add a read-only file tool that can fetch Slack-uploaded text files and photos for direct agent inspection
3. Add carefully chosen read tools such as channel history or permalink lookup
4. Keep search/discovery limited to what bot-token visibility allows
5. Add rate-limit handling and retry behavior
6. Keep the agent-driven reply path via `post_message` MCP tool (through Slack proxy)

**Exit criteria**:

- Thor can inspect user-provided attachment and photo metadata through the existing thread/channel read tools
- Thor can fetch the contents of supported Slack files directly when metadata alone is insufficient
- `post_message` remains a text-only bot reply path
- The tool surface remains intentionally small and documented

### Later Phase 2 candidates

1. Add carefully chosen read tools such as channel history or permalink lookup
2. Keep search/discovery limited to what bot-token visibility allows
3. Add rate-limit handling and retry behavior
4. Keep the agent-driven reply path via `post_message` MCP tool (through Slack proxy)

### Phase 3 — Optional Hosted Slack MCP for User-Scoped Discovery

**Goal**: Add Slack's hosted MCP only if we need broader discovery than bot membership allows.

Steps:

1. Identify concrete read scenarios the bot-token MCP cannot satisfy
2. Add hosted Slack MCP directly to OpenCode as a separate read-oriented integration
3. Keep posting and approvals on the bot-token path
4. Revisit proxy OAuth support only if policy/audit becomes mandatory for user-scoped Slack reads

**Exit criteria**:

- We have a clear reason to keep both integrations
- Hosted Slack MCP is limited to read-side discovery
- Bot-authored writes remain on the internal MCP/gateway path

## Operational Notes

- Slack's hosted MCP docs explicitly describe OAuth endpoints for Slack user tokens, and list required scopes as user-token scopes.
- Slack's broader token docs say bot tokens represent the app, are not tied to a user's identity, and are usually the best choice when the app should act independently.
- For our "send as bot" requirement, the safer fit is a custom bot-token MCP surface.
- If we later add hosted Slack MCP, treat it as a separate user-scoped read integration.

## Out of Scope

- Replacing the Slack Events API gateway with MCP
- Building a full generic wrapper for all Slack Web API methods
- Unifying bot-token and user-token Slack access under one auth model in Phase 1
